# Everybody Tennis AstrBot 插件需求文档

## 1. 文档目标

本文档用于定义 `astrbot_plugin_everybody_tennis` 的当前需求基线，服务于后续开发、维护和功能扩展。

这份文档重点回答四类问题：

- 这个插件现在要解决什么问题
- 当前版本明确支持什么、不支持什么
- 核心业务规则是什么
- 后续扩展时哪些方向是预留的，哪些假设已经被放弃

如果需求文档、技术设计文档与代码行为不一致，应以代码和实际产品决策为准，并同步修正文档。

## 2. 产品背景

群友会在群内分享网球游戏对战截图。截图里包含比赛结果和双方技术统计，但这些数据如果只停留在图片里，后续无法做稳定查询、排行和长期积累。

插件的目标是把这类截图转为结构化比赛记录，并围绕这些记录提供基础统计能力。

当前产品要解决的核心问题：

- 群聊内方便录入比赛截图
- 自动提取主要比赛字段
- 允许不完整记录进入待确认，而不是直接丢弃
- 让群友能够按游戏昵称查询战绩
- 让群里能查看基础排行

## 3. 当前版本定位

当前版本是首个可用闭环版本，目标不是功能尽可能多，而是让主链路真实可用、可维护。

当前闭环是：

```text
提交截图 -> 提取结构化数据 -> 自动入库/待确认 -> 查询战绩 -> 查看排行
```

当前版本已经明确采用的产品策略：

- 按群隔离数据，不做跨群总榜
- 按游戏昵称统计，不做平台账号绑定
- 完整记录默认自动确认，不强制所有记录人工确认
- 单打和双打数据按比赛类型隔离
- 数据库存储优先保证简单可靠，允许删库重建，不做旧结构迁移兼容

## 4. 用户角色

### 4.1 普通群成员

可进行：

- 提交比赛截图
- 取消自己提交的待确认记录
- 撤销自己自动入库的记录
- 查询指定游戏昵称的战绩
- 查看最近比赛
- 查看群排行

### 4.2 群管理员

除普通成员能力外，还应具备：

- 删除异常记录
- 处理重复录入或争议数据

### 4.3 插件维护者

负责：

- 配置多模态 provider
- 调整提示词
- 维护数据库结构和统计口径
- 扩展功能与修复问题

## 5. 核心使用场景

### 5.1 场景 A：提交比赛截图

1. 用户在群内发送 `/网球 录入` 并附带一张比赛截图。
2. 插件调用多模态模型提取结构化比赛数据。
3. 插件回显识别预览和记录号。
4. 若字段完整且胜负可判定，则记录直接入库。
5. 若字段缺失，则记录保留为 `pending`，等待用户确认或取消。

### 5.2 场景 B：查询个人战绩

1. 用户发送 `/网球 战绩 <游戏昵称>`。
2. 插件按群范围和游戏昵称聚合已确认记录。
3. 插件返回战绩和主要平均技术指标。

### 5.3 场景 C：查看最近比赛

1. 用户发送 `/网球 最近 <游戏昵称> [条数]`。
2. 插件返回该昵称最近若干场已确认比赛。

### 5.4 场景 D：查看群排行

1. 用户发送 `/网球 排行 [指标] [人数]`。
2. 插件按群范围聚合玩家统计。
3. 插件返回指定指标的排行结果。

### 5.5 场景 E：撤销或删除记录

1. 用户发现识别有误或重复录入。
2. 若记录是自己提交的，可取消 `pending` 或撤销 `confirmed`。
3. 若需要彻底排除记录，管理员或有权限的提交人可执行删除。

## 6. 当前必须满足的需求

### 6.1 录入需求

当前版本必须支持：

- 群聊内提交单张比赛截图
- 支持 `/网球 双打录入`
- 自动提取双方昵称和主要技术统计
- 生成结构化预览
- 保存原图副本
- 保存原始提取结果与标准化结果
- 对完整记录自动确认入库
- 对不完整记录进入待确认状态

当前版本不要求：

- 批量上传多张截图
- 兼容所有截图 UI 变体
- 多人模式

### 6.2 确认与取消需求

当前版本必须支持：

- 对 `pending` 记录执行确认
- 对 `pending` 记录执行取消
- 对已自动入库的 `confirmed` 记录执行撤销

当前确认规则的产品要求：

- 只有提交人本人可以确认或取消自己的记录
- 只要胜负仍可判定，就允许确认
- 不是所有缺失字段都必须阻止确认

### 6.3 删除需求

当前版本必须支持：

- 软删除记录
- 删除后统计自动忽略该记录
- 管理员可删除记录
- 提交人是否可删除由配置控制

### 6.4 查询需求

当前版本必须支持：

- `/网球 战绩 <游戏昵称>`
- `/网球 最近 <游戏昵称> [条数]`
- `/网球 排行 [指标] [人数]`

当前查询策略要求：

- 查询时显式指定游戏昵称
- 统计只在当前群内生效
- 当前单打查询和排行仅基于 `confirmed` 的 `singles` 数据
- 双打查询和双打排行后续再补

## 7. 当前产品规则

### 7.1 胜负判定规则

`winner_side` 不依赖模型输出，由系统根据双方 `points_won` 自动推导：

- 点数高的一方获胜
- 任一方点数缺失，或双方点数相同，则无法判定胜负

这条规则是当前产品设计的硬约束，后续不要改回“让模型直接决定 winner_side”。

### 7.2 自动确认规则

记录自动进入 `confirmed` 的条件：

- `missing_fields` 为空
- `winner_side` 可以判定

否则记录进入 `pending`。

双打补充规则：

- 录入命令为 `/网球 双打录入`
- 固定 4 名玩家
- 前两名为队伍1，后两名为队伍2
- 胜负按同队两人的 `points_won` 总和比较
- `发球最高球速（km/h）` 只在双打提取

### 7.3 统计主体规则

当前统计主体是游戏昵称，不是平台账号。

当前规则：

- 入库时保存原始昵称和标准化昵称
- 查询时按标准化昵称匹配
- 排行时按标准化昵称聚合

当前不做：

- 平台账号与游戏昵称绑定
- 别名合并
- 跨群统一身份

### 7.4 状态规则

当前记录状态包括：

- `pending`
- `confirmed`
- `cancelled`
- `expired`
- `deleted`

正式统计只基于 `confirmed`。

### 7.5 去重规则

当前版本只要求具备基础重复提示能力，不要求复杂判重。

当前可接受的最小能力：

- 基于图片哈希识别疑似重复
- 在录入预览中提醒用户

## 8. 数据需求

### 8.1 当前关注的字段

首版至少关注这些字段：

- 双方玩家昵称
- 点数取得次数
- 胜球次数
- 发球得分次数
- 失误球次数
- 双发失误次数
- 网前截击率
- 比赛盘数
- 比赛局数
- 比赛时长
- 最长回合

双打额外字段：

- 发球最高球速（km/h）

### 8.2 持久化要求

系统需要长期保存：

- 群信息
- 比赛记录
- 每场比赛的双方玩家统计
- 原图路径
- 图片哈希
- 原始提取 JSON
- 标准化 JSON

### 8.3 当前数据库边界

当前需求上只需要 3 类核心实体：

- 群
- 比赛
- 单场玩家统计

当前不要求维护独立的玩家主表，也不要求别名表。

## 9. 统计口径

统计口径必须保持稳定，否则历史数据将难以解释。

当前口径如下：

- 比赛场次：已确认记录数
- 胜场：`is_winner = true` 的记录数
- 负场：比赛场次减去胜场
- 胜率：胜场除以比赛场次
- 场均得分：`points_won` 总和除以比赛场次
- 场均胜球：`winners` 总和除以比赛场次
- 场均发球得分：`serve_points_won` 总和除以比赛场次
- 场均失误：`errors` 总和除以比赛场次
- 场均双误：`double_faults` 总和除以比赛场次
- 平均网前截击率：`net_play_rate` 简单平均

当前产品要求特别注意：

- 技术项展示以平均值为主，不再强调累计值
- 胜率排行需要最少比赛场次门槛
- `cancelled`、`expired`、`deleted`、`pending` 数据都不能进入正式统计

## 10. 命令需求基线

当前已固定的命令基线：

- `/网球 帮助`
- `/网球 录入`
- `/网球 确认 <记录号>`
- `/网球 取消 <记录号>`
- `/网球 删除 <记录号>`
- `/网球 战绩 <游戏昵称>`
- `/网球 最近 <游戏昵称> [条数]`
- `/网球 排行 [指标] [人数]`

如果后续要改命令名、参数格式或交互文案，需要同步更新：

- `README.md`
- `docs/technical_design.md`
- 命令帮助文本
- 回归测试

## 11. 非功能需求

### 11.1 准确性

- 模型输出异常时，必须给出明确错误提示
- 缺失字段不能伪造
- 正式统计只能使用 `confirmed` 数据

### 11.2 可维护性

- 业务逻辑不能继续堆进 `main.py`
- 配置读取、模型调用、统计聚合、消息渲染必须分层
- 新需求应尽量落在现有服务边界内扩展

### 11.3 性能

- 单次截图提取允许秒级延迟
- 群内常规规模下，查询和排行应快速返回

### 11.4 一致性

- 同一条记录不能重复确认
- 删除和取消必须有明确状态校验
- 文档、README、代码行为要保持一致

### 11.5 权限

- 默认只允许在群聊中使用
- 删除等高风险操作必须做权限控制
- 一个群的数据不能被另一个群查询

## 12. 当前不做的需求

以下方向已经明确不属于当前版本：

- 跨群总榜
- Web 管理后台
- 完整人工逐字段修正界面
- 用户身份绑定
- 昵称别名合并
- 时间范围筛选
- 周报/月报推送
- 图片榜单渲染
- 多人模式

这些能力可以后续再做，但不应反向污染当前已稳定的主链路。

## 13. 后续扩展方向

后续优先级较高的扩展方向包括：

- 时间范围筛选
- 管理员查看待确认列表
- 管理员修正识别结果
- 更细的重复记录处理
- 更丰富的排行指标
- 图片化榜单或报表
- 玩家别名映射

扩展时的原则：

- 先保留当前主链路稳定
- 先补测试，再放大功能面
- 新能力尽量以增量方式叠加，不要推翻现有数据模型

## 14. 维护者注意事项

后续维护时，优先关注这些基线不要被破坏：

- 查询入口仍然是“按游戏昵称指定”
- 胜负仍然由系统根据点数判定
- 自动确认与待确认的分界规则要保持明确
- 正式统计只基于 `confirmed`
- 文档必须及时跟代码一起更新

如果后续决定引入身份绑定、别名映射或跨群能力，应视为新阶段需求，不要在现有规则上偷偷叠加半套逻辑。

## 15. 结论

这个插件当前的本质不是“识别一张图”，而是“把群聊里的比赛截图沉淀为可长期使用的结构化比赛记录”。

因此当前需求的核心不是命令数量，而是守住这几条基线：

- 录入链路稳定
- 状态流转明确
- 统计口径稳定
- 维护边界清晰

只要这几条基线不被破坏，后续继续加筛选、修正、图表和更多排行能力都会比较顺手。
