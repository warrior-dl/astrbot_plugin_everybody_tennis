# Everybody Tennis AstrBot 插件需求文档

## 1. 文档目标

本文档用于明确 `astrbot_plugin_everybody_tennis` 的首版需求范围，作为后续插件架构设计、数据库设计、命令设计和开发排期的基线。

当前目标不是直接实现全部能力，而是先把核心业务流程、关键数据结构、统计口径和待确认问题整理清楚。

## 2. 项目背景

群友会在群内进行网球游戏对战，并生成包含比赛结果和双方技术统计的截图。希望通过 AstrBot 插件完成以下事情：

- 用户通过指令提交对战截图。
- 机器人借助 LLM 多模态能力从截图中提取结构化比赛数据。
- 将结构化结果持久化保存，支持后续查询与统计。
- 用户可查询个人战绩、个人统计和群排行。

从当前示例截图文字可确认，首版至少会涉及以下数据字段：

- 比赛盘数、局数
- 双方玩家昵称
- 点数取得次数
- 胜球次数
- 发球得分次数
- 失误球次数
- 双发失误次数
- 网前截击率
- 最多对打次数
- 比赛时间

## 3. 目标与非目标

### 3.1 目标

- 打通“截图提交 -> 多模态提取 -> 用户确认 -> 数据入库 -> 统计查询”的完整链路。
- 在群维度沉淀可长期积累的比赛数据。
- 提供个人战绩、个人统计和群排行三类核心查询能力。
- 为后续扩展更多统计维度、报表和可视化能力预留空间。

### 3.2 非目标

- 首版不追求覆盖所有可能的截图样式和所有比赛模式。
- 首版不要求复杂 Web 管理后台。
- 首版不要求跨群全局合并排行榜，默认以群为统计边界。
- 首版不要求完全脱离 LLM 的传统 OCR/视觉模型流程。

## 4. 角色与使用场景

### 4.1 用户角色

- 普通群成员：提交比赛截图，查询自己的比赛记录和统计结果。
- 群管理员或插件管理员：查看群排行，必要时删除异常数据或处理争议记录。
- 插件维护者：配置模型、数据库、统计规则和权限策略。

### 4.2 典型使用场景

#### 场景 A：提交比赛结果

1. 用户在群内发送录入指令，并附带一张比赛截图。
2. 插件调用多模态模型识别截图内容。
3. 插件将识别出的结构化数据回显给用户。
4. 用户确认后，记录正式入库。

#### 场景 B：查询个人战绩

1. 用户发送查询指令。
2. 插件按群维度检索该用户的历史比赛。
3. 插件返回比赛场次、胜负、胜率和主要技术统计。

#### 场景 C：查看群排行

1. 用户发送排行指令。
2. 插件按指定指标和时间范围统计群内玩家数据。
3. 插件返回排行榜结果。

#### 场景 D：纠错或删除

1. 用户发现识别错误或重复录入。
2. 插件允许用户取消未确认记录，或让有权限的人删除已入库记录。

## 5. 核心业务流程

### 5.1 数据录入流程

建议首版采用“先识别、后确认、再入库”的流程：

1. 用户发送录入指令，并附带截图。
2. 插件校验消息上下文：
   - 是否在群聊中使用
   - 是否包含图片
   - 是否命中受支持的截图类型
3. 插件调用当前会话绑定的多模态 LLM，提取结构化字段。
4. 插件生成一条待确认记录，并把识别结果展示给用户。
5. 用户通过确认指令完成入库；或取消本次录入。
6. 插件将正式记录写入持久化数据库。
7. 插件更新相关统计缓存或在查询时动态聚合。

之所以要求确认环节，是因为多模态识别存在误识别、字段漏提取和昵称不一致风险。如果直接入库，后续统计可信度会快速下降。

### 5.2 查询流程

1. 用户发送查询指令。
2. 插件识别查询对象、时间范围和统计指标。
3. 插件从数据库聚合数据。
4. 插件返回文本结果，后续可扩展为图片榜单或图表。

## 6. 功能需求

### 6.1 比赛截图录入

首版必须支持：

- 在群聊中通过指令提交比赛截图。
- 自动识别截图中的双方玩家和比赛统计数据。
- 对识别结果生成结构化预览。
- 支持用户确认或取消。
- 对确认后的比赛记录正式入库。

首版建议支持：

- 记录截图原图地址或本地副本路径，方便审计和问题排查。
- 保存 LLM 原始提取结果和标准化结果，便于后续优化提示词。
- 记录提交人、提交时间、群号、平台来源。

### 6.2 数据纠错与删除

首版必须支持：

- 取消尚未确认的待入库记录。
- 删除已入库比赛记录，至少允许提交人本人或管理员操作。

首版建议支持：

- 查看最近几条录入记录。
- 为误识别场景保留人工修正入口，哪怕首版只是管理员文本修正。

### 6.3 个人查询

首版必须支持：

- 查询个人总比赛场次。
- 查询个人胜场、负场、胜率。
- 查询个人累计技术统计，例如总得分、总胜球、总发球得分、总失误、总双误。
- 查询个人最近比赛记录。

首版建议支持：

- 查询某个时间范围内的数据，例如近 7 天、本月、全部。
- 支持查询自己，也支持管理员或普通用户查询被 @ 的玩家数据。

### 6.4 群排行

首版必须支持：

- 群内总胜场排行。
- 群内胜率排行。
- 群内比赛场次排行。

首版建议支持：

- 按技术指标排行，例如累计得分、累计胜球、累计发球得分、最少双误。
- 按时间范围排行，例如本周、本月、全部。
- 设置最低参赛场次门槛，避免只打 1 场就占据胜率榜首。

### 6.5 管理能力

首版建议支持：

- 管理员查看待确认记录或异常记录。
- 管理员删除重复数据。
- 管理员配置是否仅允许群内使用。

## 7. 命令草案

以下是建议的首版命令，不是最终实现名，但建议尽早固定：

- `/网球录入`：提交截图并创建待确认记录。
- `/网球确认 <记录ID>`：确认本次识别结果并入库。
- `/网球取消 <记录ID>`：取消本次待确认记录。
- `/网球战绩 [@用户] [时间范围]`：查看个人战绩汇总。
- `/网球最近 [@用户]`：查看最近比赛记录。
- `/网球排行 [指标] [时间范围]`：查看群排行。
- `/网球删除 <记录ID>`：删除已入库记录。
- `/网球帮助`：查看插件支持的命令和说明。

后续如果 AstrBot 消息交互体验允许，也可以把“确认/取消”做成更自然的会话式交互，但底层仍应对应明确的记录状态流转。

## 8. 数据需求与数据模型草案

### 8.1 为什么需要数据库

该插件的核心能力是长期积累比赛数据并进行聚合统计，因此不适合只用简单 KV 存储。首版更适合使用关系型数据库。

推荐首版方案：

- 数据库存储：SQLite
- 文件位置：`data/plugin_data/astrbot_plugin_everybody_tennis/tennis.db`

原因：

- 查询、排行、筛选、聚合需求明确。
- 数据规模在群插件场景下通常不大，SQLite 足够支撑首版。
- 后续可平滑迁移到 PostgreSQL 或 MySQL。

### 8.2 核心实体

建议至少包含以下实体：

### Group

- `group_id`
- `platform`
- `group_name`
- `created_at`

### Player

- `player_id`
- `group_id`
- `platform_user_id`
- `display_name`
- `alias_name`
- `created_at`
- `updated_at`

说明：

- 首版建议按“群内玩家”建模，而不是做跨群统一身份。
- 同一个人在不同群可视为不同统计主体，避免身份映射复杂化。

### Match

- `match_id`
- `group_id`
- `submitted_by`
- `status`：`pending` / `confirmed` / `cancelled` / `deleted`
- `source_image_path`
- `source_image_url`
- `raw_extraction_text`
- `normalized_json`
- `match_date`
- `set_count`
- `game_count`
- `duration_seconds`
- `max_rally_count`
- `winner_player_id`
- `loser_player_id`
- `confirmed_at`
- `created_at`
- `updated_at`

### MatchPlayerStat

- `id`
- `match_id`
- `player_id`
- `is_winner`
- `points_won`
- `winners`
- `serve_points_won`
- `errors`
- `double_faults`
- `net_play_rate`

说明：

- 比赛公共字段和球员个人字段要拆开，便于统计和扩展。
- `net_play_rate` 建议保存为标准化数值，例如 `0.466`，不要直接存字符串百分比。

### ExtractionLog

- `id`
- `match_id`
- `provider_id`
- `model_name`
- `prompt_version`
- `raw_response`
- `parse_success`
- `confidence_score`
- `created_at`

说明：

- 这不是用户可见实体，但对后续调试多模态提取质量很重要。

## 9. 统计口径

首版必须尽早统一统计定义，避免后续口径混乱。

建议口径如下：

- 比赛场次：已确认且未删除的记录数。
- 胜场：`is_winner = true` 的比赛数。
- 负场：`比赛场次 - 胜场`。
- 胜率：`胜场 / 比赛场次`。
- 累计得分：`points_won` 求和。
- 累计胜球：`winners` 求和。
- 累计发球得分：`serve_points_won` 求和。
- 累计失误：`errors` 求和。
- 累计双误：`double_faults` 求和。
- 平均网前截击率：对单场 `net_play_rate` 做平均，或后续补充更科学的加权口径。

需要特别注意：

- 胜率排行应设置最少比赛场次门槛，例如至少 3 场。
- 群排行默认只统计当前群的数据。
- 被删除、取消、未确认的数据不能进入正式统计。

## 10. 多模态提取需求

### 10.1 输入要求

- 支持用户上传单张比赛结果截图。
- 首版先以当前示例样式为主，不强行兼容所有 UI 风格。
- 必须允许后续扩展不同截图模板的识别策略。

### 10.2 输出要求

多模态提取阶段应输出标准 JSON，至少包含：

- 双方玩家名称
- 比赛盘数/局数
- 双方技术统计字段
- 比赛时长
- 最长回合
- 识别是否完整
- 缺失字段列表

### 10.3 稳定性要求

- 识别失败时要明确告知原因，而不是静默失败。
- 字段缺失时要允许用户取消，不应强制入库。
- 对名称识别要做标准化处理，例如去空格、统一大小写、保留别名能力。

## 11. 非功能需求

### 11.1 准确性

- 正式统计只能基于用户确认后的数据。
- 系统需要保留原始截图和提取日志，以便排查错误。

### 11.2 可维护性

- 插件应采用工程化分层结构，不建议把全部逻辑塞进 `main.py`。
- 配置、识别编排、统计服务、数据访问层应分离。

### 11.3 性能

- 单次截图识别允许有数秒级延迟，但必须给用户明确反馈。
- 查询和排行应在正常群规模下快速返回。

### 11.4 并发与一致性

- 同一截图被重复提交时，系统应具备基本的去重或提示能力。
- 同一条待确认记录不能被重复确认。
- 删除和确认操作应具备状态校验。

### 11.5 安全与权限

- 默认仅允许在群内录入和查看群统计。
- 删除等高风险操作需要权限控制。
- 不应把一个群的数据暴露给另一个群。

## 12. 首版范围建议

建议将 MVP 范围控制在以下能力：

- 群内截图录入
- 多模态提取
- 识别结果确认/取消
- SQLite 持久化
- 个人战绩查询
- 最近比赛查询
- 群内基础排行
- 管理员删除记录

先不纳入 MVP 的能力：

- 跨群总榜
- 富文本后台管理页面
- 完整人工逐字段修正界面
- 自动识别多张截图并批量入库
- 复杂图表分析与周报推送

## 13. 后续版本方向

- 文本转图片生成更美观的排行榜卡片。
- 增加月榜、周榜、赛季榜。
- 支持玩家别名合并和身份映射。
- 支持导出 CSV 或 JSON。
- 支持更丰富的技术统计和趋势分析。
- 支持管理员修正历史记录。

## 14. 当前待确认问题

这些问题会直接影响设计，建议在开发前尽快确认：

1. 一场比赛是否固定只有两名玩家，还是后续会出现双打或特殊模式。
2. 当前截图来源是否固定为同一款游戏、同一套界面样式。
3. 比赛时间是否需要单独记录真实对战日期，而不是仅使用提交时间。
4. 玩家身份是否以群昵称为准，还是允许绑定 QQ 号/平台用户 ID。
5. 同一场比赛如果双方都提交截图，是否视为重复记录，需要怎样的去重规则。
6. 管理员是否需要人工修正识别结果，而不是只能确认/取消。
7. 排行榜最终最看重哪些指标，是否需要提前锁定展示顺序。

## 15. 结论

这个插件的本质不是“识别一张图”，而是“沉淀一套可信的群内网球对战数据库，并围绕它提供持续统计能力”。因此首版最关键的不是命令数量，而是先把以下三件事做扎实：

- 可靠的结构化提取
- 明确的确认入库流程
- 稳定的统计口径与数据库模型

只要这三层定稳，后续再补排行样式、图表和更多高级功能会顺畅很多。
